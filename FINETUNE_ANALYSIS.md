# Анализ поведения Qwen3 8B в контент-фабрике SciWriter MAS

## 1. Резюме

Проанализировано 16 статей, сгенерированных моделью Qwen3 8B в пайплайне SciWriter MAS. Из них 12 статей основаны на одном источнике (Tesla MCU teardown), 1 на теме контрольных комнат (bench_15), 1 на теме CERN/ORE (bench_16), 1 на теме Mac Pro (bench_17), и 1 — тестовая статья про ИИ в науке (test_habr).

**Ключевые выводы:**

- **Галлюцинации присутствуют в 100% статей** — ни одна статья не свободна от выдуманных фактов
- **Самый частый тип** — fabricated numbers (выдуманные числа/характеристики), обнаружен в 14 из 16 статей
- **Модель систематически "дописывает" источник** — добавляет технические детали, которых нет в оригинале (характеристики процессоров, размеры в мм, FPS, ватты)
- **Ссылки фиктивные в 100% случаев** — все ссылки ведут на один URL источника или на example.com, при этом приписываются конкретным фактам, которых в источнике нет
- **Rater output просачивается в текст** — bench_14 целиком является выводом rater, а не статьей
- **Пустые секции** — встречаются в bench_14 (раздел "Питание и охлаждение" пуст)
- **SEO-мусор** — Keywords, Meta Description, Changelog в теле статьи (article_2, article_4, test_habr)
- **Кириллица в URL** — обнаружена в article_4 (буква "к" вместо "k" в URL)

Модель требует fine-tune для подавления галлюцинаций, обучения отказу при отсутствии данных, и исключения артефактов пайплайна из финального текста.

## 2. Методология анализа

### Источники
- **Эталон (ground truth):** `source_original.txt` — оригинальная статья автора (bugs.xdavidhu.me) про запуск компьютера Tesla Model 3 на столе
- **Статьи Tesla-темы (12 шт.):** bench_14, article_2, article_4, article_5, article_v5, article_v5_final, article_v6, article_v7, article_latest_v4, article_6, article_latest, article_v8
- **Статьи на другие темы (4 шт.):** bench_15 (контрольные комнаты), bench_16 (CERN/ORE), bench_17 (Mac Pro), test_habr (ИИ в науке)

### Процедура
1. Каждая статья прочитана полностью
2. Каждое фактическое утверждение сверено с оригинальным источником
3. Утверждения классифицированы как: CORRECT (подтверждено источником), HALLUCINATED (отсутствует в источнике), FABRICATED NUMBER (выдуманная цифра)
4. Отмечены структурные проблемы, артефакты пайплайна, проблемы качества текста

### Факты из оригинального источника (ground truth для Tesla-статей)
- MCU + AP — два модуля, расположены друг на друге
- Размер — как iPad, толщина — как книга ~500 страниц
- Корпус — металлический с водяным охлаждением
- MCU на eBay: $200-$300
- Экран: $175
- Блок питания: 0-30V, версии 5A и 10A, выбрал 10A
- Пиковое потребление: до 8A
- Кабель дисплея: 6-пиновый, Rosenberger 99K10D-1D5A5-D
- BMW LVDS кабель не подошел (разъем толще)
- Замыкание повредило чип MAX16932CATIS/V+T
- Пришлось купить второй компьютер
- Друг Yasser (@n3r0li) опознал чип
- Починили в PCB repair shop
- Купили целый Dashboard Wiring Harness за $80
- Tesla публикует Electrical Reference (схемы)
- Ethernet порт, сеть 192.168.90.X/24, без DHCP
- SSH на порту 22, ODIN API на :8080
- Bug bounty программа Tesla, Root access program
- Мотивация автора — участие в bug bounty

## 4. Классификация типов галлюцинаций

### 4.1. Фабрикация технических характеристик (FABRICATED SPECS)
**Частота: 14 из 16 статей**

Модель выдумывает конкретные числовые характеристики, которых нет в источнике.

**Примеры:**
- bench_14: "процессор NVIDIA Tegra с частотой 2.4 ГГц, 4 ГБ RAM, 16 ГБ NAND" — ничего из этого нет в источнике
- bench_14: "2 ТБ SSD с интерфейсом NVMe" — полностью выдумано
- article_5: "MCU: 120 x 80 мм, вес 150 г, частота 1.8 ГГц, 2 ГБ RAM" — ни одной из этих цифр нет в источнике
- article_5: "AP: 2.4 ГГц, 4 ГБ RAM, 12-канальный видеовыход" — выдумано
- article_5: "Запуск системы: 3 секунды, 120 FPS, 15 Вт" — полностью фабрикованные метрики
- article_v5_final: "Размеры: 200x150 мм (MCU), 300x200 мм (экран)" — выдумано
- article_v5_final: "Вес: 500 г (MCU), 1 кг (экран)" — выдумано
- article_latest: "28 см x 18 см, толщина 5 см" — в источнике сказано "размер iPad, толщина ~500 стр. книги", точных размеров нет
- article_latest: "MIPI DSI, 1.5 Гбит/с, 120 Гц" — кабель в источнике Rosenberger, протокол не указан
- article_v7: "300-500 Вт блок питания" — в источнике 0-30V/10A (макс ~300 Вт при полной нагрузке, но так не описано)
- article_latest_v4: "50 млн тонн электронного мусора по данным Greenpeace" — данные могут быть реальными, но не из источника

### 4.2. Фабрикация цен (FABRICATED PRICES)
**Частота: 8 из 16 статей**

Модель выдумывает цены, искажая данные источника.

**Примеры:**
- article_5: "MCU: $120, AP: $180, кабель: $30, БП: $50" — в источнике MCU $200-300, экран $175, harness $80
- article_latest_v4: "MCU ~$150, AP ~$300, плата PCIe $50" — AP вообще не продавался отдельно в источнике
- article_v7: "AP: $300-$500, тачскрин: $50-$100" — в источнике экран $175, AP не указан отдельно
- article_v6: "от $50 до $200" — занижено, MCU $200-300 в источнике
- article_v6: "кабели 14 AWG" — в источнике нет упоминания калибра проводов
- article_6: "AP: ~$500, кабель: ~$150" — AP и кабель не продавались отдельно (купили harness за $80)

### 4.3. Искажение нарратива (NARRATIVE DISTORTION)
**Частота: 12 из 16 статей**

Модель меняет мотивацию и контекст проекта.

**Примеры:**
- Все Tesla-статьи: мотивация переписана как "переработка электроники", "экология", "устойчивое развитие" — в источнике мотивация: **участие в bug bounty программе Tesla**
- article_5: "снижение отходов на 70%" — полностью выдуманная статистика
- bench_14: "снижение стоимости на 40%" без источника — повторяется трижды в тексте
- article_2: "FSD-модуль (Full Self-Driving)" — в источнике речь о MCU+AP, FSD не упоминается
- article_2: "модуль находится в задней части салона" — в источнике: перед пассажирским сиденьем, за бардачком
- article_2: "скорость обработки данных, сравнимая с современными GPU" — выдуманное утверждение
- article_v5: "MCU: извлекли из блока управления тормозами" — MCU не имеет отношения к тормозам
- article_v6: "В 2023 году группа хакеров из Калифорнии..." — полностью выдуманный пример

## 5. Анализ работы валидатора

### Что валидатор ловит
На основании содержимого статей можно судить, что валидатор (если он есть) ловит:
- Некоторые грубые фактические ошибки (в поздних версиях article_v5 -> v5_final -> v6 -> v7 видна итеративная правка)
- Дублирование разделов (bench_14 содержит комментарий о "повторяющейся информации")

### Что валидатор пропускает
- **100% фабрикованных чисел** — все выдуманные характеристики (размеры в мм, вес, FPS, ватты) проходят через валидацию
- **100% искаженных нарративов** — замена мотивации (bug bounty -> экология) не отлавливается
- **100% фейковых процедур** — выдуманные технические шаги не проверяются
- **100% фейковых ссылок** — валидатор не проверяет, что факт действительно содержится по указанной ссылке
- **Артефакты пайплайна** — FACT-метки, rater output в тексте, SEO-блоки не удаляются
- **Кириллица в URL** — не детектируется
- **Пустые секции** — bench_14 имеет пустой раздел "Питание и охлаждение"

### Эволюция версий (Tesla-статьи)
Сравнение версий показывает минимальное улучшение:
- article_2 -> article_4: добавлен Python-код (не из источника), появился FSD (галлюцинация)
- article_4 -> article_5: размер уменьшился, но галлюцинации усилились (120 FPS, 15 Вт)
- article_v5 -> v5_final -> v6 -> v7: добавляются детали, но каждая новая деталь — галлюцинация
- article_latest_v4: FACT-метки в тексте = артефакт пайплайна не удален

**Вывод:** Итеративная переработка не уменьшает галлюцинации, а наоборот — каждая итерация добавляет новые выдуманные детали.

## 6. Анализ работы Rater

### bench_14 — Rater output вместо статьи
Файл bench_14.md начинается с блока "Причины:" и содержит оценку статьи с разбором структуры, а затем "Улучшенную версию". Это означает, что:
1. Rater генерирует полноценный ответ с оценкой + переписанной версией
2. Его output попадает в финальный файл без фильтрации
3. Переписанная версия содержит ВСЕ те же галлюцинации + новые

### Калибровка оценок
На основании bench_14 (единственный файл с видимым rater output):
- Rater отмечает "повторяющуюся информацию" — корректно
- Rater отмечает "отсутствуют ссылки на источники" — корректно
- Rater отмечает "40% без ссылок снижает доверие" — корректно
- Rater НЕ отмечает выдуманные характеристики процессора (NVIDIA Tegra 2.4 ГГц, 4 ГБ RAM)
- Rater НЕ отмечает выдуманный "2 ТБ SSD NVMe"
- Rater НЕ отмечает пустую секцию "Питание и охлаждение"

### Проблемы калибровки
1. **Rater не сверяет факты с источником** — он оценивает "правдоподобность" текста, а не соответствие источнику
2. **Rater фокусируется на стиле, а не на содержании** — отмечает "мог бы быть более живым", но пропускает грубые фактические ошибки
3. **Rater генерирует новые галлюцинации** — "улучшенная версия" в bench_14 содержит те же выдуманные цифры
4. **Нет числовой шкалы** — rater дает текстовый фидбек без score, что затрудняет автоматизацию

## 7. Цели fine-tune

### 7.1. Подавление фабрикации чисел
**Текущее состояние:** Модель выдумывает конкретные числа в 88% статей
**Цель:** Снизить до <5% статей с фабрикованными числами
**Метрика:** % статей, содержащих числовые утверждения, отсутствующие в источнике
**Методика:** Обучить модель выводить только числа, явно присутствующие в source, или писать "данные недоступны"

### 7.2. Подавление искажения нарратива
**Текущее состояние:** Мотивация автора подменяется в 100% Tesla-статей
**Цель:** Корректная передача мотивации и контекста в >90% случаев
**Метрика:** % статей, где центральная мотивация автора передана верно
**Методика:** В датасете подчеркивать мотивацию/контекст как обязательный элемент

### 7.3. Исключение фейковых технических процедур
**Текущее состояние:** 63% статей содержат выдуманные процедуры
**Цель:** 0% выдуманных процедур
**Метрика:** Наличие шагов/команд/микросхем, отсутствующих в источнике
**Методика:** Обучить на примерах "если процедура не описана в источнике — не описывай"

### 7.4. Честные ссылки
**Текущее состояние:** 100% статей содержат фейковые или неинформативные ссылки
**Цель:** Каждая ссылка ведет на конкретный факт, который действительно присутствует в источнике
**Метрика:** % ссылок, где факт действительно подтверждается по указанному URL
**Методика:** Обучить модель не расставлять ссылки на факты, которые она сама генерирует

### 7.5. Удаление артефактов пайплайна
**Текущее состояние:** 31% статей содержат FACT-метки, rater output, SEO-блоки
**Цель:** 0% артефактов в финальном тексте
**Метрика:** Наличие паттернов FACT-N, Keywords:, Meta Description, Changelog в тексте
**Методика:** Post-processing или обучение на "чистых" примерах

### 7.6. Сохранение уникальных деталей
**Текущее состояние:** Ключевые детали (Rosenberger, BMW LVDS, MAX16932, bug bounty) пропущены в ~100% статей
**Цель:** Сохранение >80% уникальных деталей источника
**Метрика:** Покрытие ключевых фактов источника в статье
**Методика:** В датасете включить примеры, где уникальные детали сохранены

### 7.7. Отказ при отсутствии данных
**Текущее состояние:** Модель никогда не пишет "данные недоступны" — она всегда генерирует что-то
**Цель:** При отсутствии данных модель явно указывает это
**Метрика:** Наличие явных маркеров "нет данных" вместо галлюцинаций
**Методика:** Включить в датасет примеры с пропусками и корректными отказами

## 8. Рекомендации по датасету для fine-tune

### 8.1. Структура датасета
Формат: JSONL, каждая строка содержит:
```json
{
  "source_text": "оригинальный текст/статья",
  "source_facts": ["факт 1", "факт 2", ...],
  "good_article": "эталонная статья на основе source_text",
  "bad_article": "статья с типичными галлюцинациями (негативный пример)",
  "hallucination_labels": [{"text": "...", "type": "fabricated_number", "span": [10, 25]}]
}
```

### 8.2. Категории примеров

**Позитивные примеры (40% датасета):**
- Статья корректно передает факты из источника
- Числа только из источника
- Мотивация автора сохранена
- Уникальные детали (имена, модели, цены) присутствуют
- При отсутствии данных явно написано "не указано" или раздел опущен
- Нет SEO-блоков, FACT-меток, rater output

**Негативные примеры с разметкой (40% датасета):**
- Взять реальные статьи из текущего output
- Разметить каждую галлюцинацию с типом и span
- Показать модели, что именно является ошибкой

**Граничные случаи (20% датасета):**
- Источник с минимумом фактов — модель должна написать короткую статью, а не дописывать
- Источник с числами — модель должна использовать только эти числа
- Источник без мотивации — модель не должна придумывать мотивацию
- Несколько источников — модель должна различать факты из разных источников

### 8.3. Объем датасета
- **Минимальный:** 200 пар (source, article) с разметкой
- **Рекомендуемый:** 500-1000 пар
- **Источники для сбора:** HackerNews статьи, ArXiv абстракты, техноблоги
- **Языки:** русский (основной), можно добавить английский для robustness

### 8.4. Аугментация
- Парафраз источника (те же факты, другие слова)
- Удаление фактов из источника (тест на то, что модель не дописывает)
- Добавление отвлекающих тем (тест на то, что модель не уходит в off-topic)

## 9. Метрики успеха

### 9.1. Метрика точности фактов (Fact Precision)
```
Fact_Precision = (число верных фактов в статье) / (общее число фактических утверждений в статье)
```
- **Текущее:** ~25-35% (в среднем 3-4 верных из 11-15 утверждений)
- **Цель после fine-tune:** >85%

### 9.2. Метрика покрытия фактов (Fact Recall)
```
Fact_Recall = (число фактов источника в статье) / (общее число фактов в источнике)
```
- **Текущее:** ~15-20% (3-5 из ~25 ключевых фактов источника)
- **Цель после fine-tune:** >60%

### 9.3. Метрика галлюцинаций (Hallucination Rate)
```
Hallucination_Rate = (число галлюцинаций) / (общее число фактических утверждений)
```
- **Текущее:** ~65-75%
- **Цель после fine-tune:** <15%

### 9.4. Метрика чистоты (Artifact Rate)
```
Artifact_Rate = (число статей с артефактами пайплайна) / (общее число статей)
```
- **Текущее:** 31%
- **Цель после fine-tune:** 0%

### 9.5. Метрика качества ссылок (Reference Quality)
```
Ref_Quality = (число ссылок, подтверждающих факт) / (общее число ссылок)
```
- **Текущее:** ~10% (только цены MCU и экрана из 74 ссылок)
- **Цель после fine-tune:** >80%

### 9.6. Автоматизация оценки
Для автоматизации рекомендуется:
1. **NLI-модель** (Natural Language Inference) для проверки: "утверждение X следует из источника Y?" — deberta-v3-large или аналог
2. **Regex-детектор артефактов:** паттерны FACT-\d+, Keywords:, Meta Description, Changelog, example\.com
3. **URL-валидатор:** проверка на кириллицу в URL, дубликаты ссылок, ссылки на example.com
4. **Числовой экстрактор:** сравнение чисел в статье с числами в источнике

### 9.7. Бенчмарк-набор
Сохранить текущие 16 статей как baseline бенчмарк. После fine-tune прогнать те же источники через обновленную модель и сравнить метрики.

| Метрика | Baseline (Qwen3 8B) | Цель |
|---|---|---|
| Fact Precision | ~30% | >85% |
| Fact Recall | ~18% | >60% |
| Hallucination Rate | ~70% | <15% |
| Artifact Rate | 31% | 0% |
| Reference Quality | ~10% | >80% |
